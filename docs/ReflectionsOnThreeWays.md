# Reflections on Three Ways

Our Reflections on how much we as a group adhere to the Three Ways.

## First Way - The Principles of Flow

Make work visible -> we have pretty kanban board

Limit work in process (WIP) -> We are not really explicitly addressing this one (other than the github project board warnings). We could consider adding structure for it.

Reduce batch size -> We try to keep our tasks small

Reduce the number of handoffs -> We good (We should however be aware of not creating knowledge silos)

Continually Identify and elevate our constraints -> We have a lot of work to do since the system is unstable but this should be a priority to elevate psychological safety

Eliminate hardships and waste in the value stream -> We are setting up automations/workflows (or at least plan to). We generally do not have that much waste.

## Second Way - The Principles of Feedback

We are good at swarming problems. We have set up automatic workflows for ensuring quality. We are all responsible for ensuring quality and give each other feedback - e.g. through our PRs. It usually takes less than a day before the code is reviewed and feedback is given.

## Third Way - The Principles of Continual learning & Experimentation

Safety Culture -> We aim for high psycological safety. We are here to learn. We are new to this. We all want a good time. We are good at asking the other team members when we need help with something and have a space where we can admit when we don't know how to continue with our tasks.

Enable and inject learning into daily work -> Yes, we have implemented many technologies none of us have worked with before like DigitalOcean, Dagger, Vagrant and so on. A large part of our time spent on tasks revolves learning and keeping up with changes in miniTwit.

Convert local discoveries into global improvements -> Not really we don't have much time to talk with each other so many things will not be comunicated to other members directly. Usually we will write discoveries in the PR description so that others can see them.

Reserve Time to Create Organizational Learning and Improvement -> Yeah. We don't do that.

Information Security as Everyone’s Job, Every Day -> We don't really do a lot of security besides hashing passwords and we never store secrets in plaintext. (however no SSL or TLS makes eavesdropping attacks very easy)

Protecting the Deployment Pipeline -> What? No keys in the yaml? Because then yes

Do you learn from failures without blame? -> Yes we are good at talking with each other but in a kind and good way despite the system being on the verge of dying
